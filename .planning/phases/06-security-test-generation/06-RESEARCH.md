## Summary

Phase 5 shipped functional but minimal security and test generation. The Jinja2 renderer produces `security/security.xml` (module category + User/Manager groups with `implied_ids`) and `security/ir.model.access.csv` (User row: RWC, Manager row: RWCU). The `test_model.py.j2` template produces create + read + name_get tests. The `odoo-test-gen` agent (Phase 5 scope) adds computed/constraint/onchange tests.

Phase 6 has three concrete gaps: (1) the `security_group.xml.j2` template has a bug — the Manager `implied_ids` uses `eval="[(4, ref('group_{{ module_name }}_user'))]"` which is correct, but the template variable `module_name` should be `module_technical_name` per the module_context dict; (2) multi-company record rules are not generated at all even when `company_id` field is present on the model; (3) test coverage is shallow — write, unlink, access rights, and workflow tests are entirely missing from the Jinja2-generated stubs and the `odoo-test-gen` agent.

**Primary recommendation:** Extend the existing templates and expand `odoo-test-gen` and `odoo-security-gen` scopes. Do not hand-roll an access-rights testing framework — the existing `with_user()` + `AccessError` ORM pattern from `knowledge/testing.md` is the complete solution.

### Pattern 2: Multi-Company Record Rule Generation

**What:** When any model in the spec has a field named `company_id` of type `Many2one` with `comodel_name: "res.company"`, generate a record rule in `security.xml`.

**Trigger detection (in renderer or agent):** Check each model's fields for `company_id` with `comodel_name == "res.company"`.

**Template addition to `security_group.xml.j2`:**
```xml
{# Add after group definitions, one block per model with company_id #}
{% for model in models %}
{% set has_company = model.fields | selectattr('name', 'equalto', 'company_id') | list %}
{% if has_company %}
    <!-- Multi-company record rule for {{ model.name }} -->
    <record id="{{ model.name | to_xml_id }}_company_rule" model="ir.rule">
        <field name="name">{{ model.name }}: Multi-Company</field>
        <field name="model_id" ref="{{ model.name | model_ref }}"/>
        <field name="domain_force">[
            '|',
            ('company_id', '=', False),
            ('company_id', 'in', company_ids),
        ]</field>
        <field name="global" eval="True"/>
    </record>
{% endif %}
{% endfor %}
```

**Why `company_ids` not `user.company_ids.ids`:** In Odoo 17.0 record rule evaluation context, `company_ids` is a pre-evaluated list (not a record set). Using `user.company_ids.ids` also works but `company_ids` is the documented shorthand and OCA standard.

**Confidence:** HIGH — pattern verified from `knowledge/security.md` which documents this exact domain and the reasoning.

### Pattern 3: ir.model.access.csv Column Order and Format

**Confirmed column order (do not change):**
```csv
id,name,model_id:id,group_id:id,perm_read,perm_write,perm_create,perm_unlink
```

**Model reference formula:** `model_` + model name with dots replaced by underscores.
- `library.book` → `model_library_book`
- `equipment.work.order` → `model_equipment_work_order`

**Group reference formula (with module prefix):** `{module_name}.group_{module_name}_{role}`
- `library.group_library_user` (for module named `library`)

**Standard OCA permission pattern:**
| Row | perm_read | perm_write | perm_create | perm_unlink |
|-----|-----------|------------|-------------|-------------|
| User | 1 | 1 | 1 | 0 |
| Manager | 1 | 1 | 1 | 1 |

**Existing `access_csv.j2` template is correct** — it already uses `model_ref` filter (which applies the dot→underscore conversion) and includes both User and Manager rows.

**Confidence:** HIGH — verified from `knowledge/security.md` column reference table and confirmed against existing template.

### Pattern 4: Security Group `implied_ids` Syntax

**Correct XML syntax for `implied_ids`:**
```xml
<!-- User group: inherits from base internal user group -->
<record id="group_{module}_user" model="res.groups">
    <field name="name">User</field>
    <field name="category_id" ref="module_category_{module}"/>
    <field name="implied_ids" eval="[(4, ref('base.group_user'))]"/>
</record>

<!-- Manager group: inherits from module User group -->
<record id="group_{module}_manager" model="res.groups">
    <field name="name">Manager</field>
    <field name="category_id" ref="module_category_{module}"/>
    <field name="implied_ids" eval="[(4, ref('group_{module}_user'))]"/>
    <field name="users" eval="[(4, ref('base.user_root')), (4, ref('base.user_admin'))]"/>
</record>
```

**Why `(4, ref(...))` not plain `ref(...)`:** Command tuple `(4, id)` means "link this existing record". The `implied_ids` field is a Many2many — you must use ORM command tuples in XML `eval=` attributes. Using bare `ref(...)` would attempt to set the field to a single integer, not a command list.

**Confidence:** HIGH — existing `security_group.xml.j2` uses this pattern and it passes validation in Phase 5 tests.

### Pattern 5: Module Category Definition

**What:** `ir.module.category` must be defined BEFORE the `res.groups` records that reference it via `ref`.

**Template order in `security_group.xml.j2` (correct, matches existing template):**
```xml
<odoo>
    <!-- 1. Module Category (FIRST - groups reference it) -->
    <record id="module_category_{module}" model="ir.module.category">
        <field name="name">{Module Title}</field>
        <field name="sequence">100</field>
    </record>

    <!-- 2. User Group -->
    <record id="group_{module}_user" model="res.groups">
        ...
        <field name="category_id" ref="module_category_{module}"/>
        ...
    </record>

    <!-- 3. Manager Group -->
    <record id="group_{module}_manager" model="res.groups">
        ...
        <field name="category_id" ref="module_category_{module}"/>
        ...
    </record>

    <!-- 4. Record Rules (LAST - reference model external IDs) -->
    ...
</odoo>
```

**`sequence` value:** Use `100` for custom modules. Core Odoo uses 1-20, OCA modules use 70-150. `100` is OCA convention and avoids collisions.

**Confidence:** HIGH — existing template already uses this pattern; sequence=100 is consistent with knowledge/security.md.

### Pattern 6: CRUD Test Expansion

**What:** The existing `test_model.py.j2` generates `test_create` and `test_read` stubs with minimal assertions. Phase 6 must expand these stubs and add `test_write` and `test_unlink`.

**What the odoo-test-gen agent must add (in addition to Phase 5 computed/constraint/onchange):**

```python
# Source: knowledge/testing.md CRUD Tests section

def test_write_{model_var}(self):
    """Test that {model_description} record can be updated."""
    self.{model_var}.write({"name": "Updated Name"})
    self.assertEqual(self.{model_var}.name, "Updated Name")

def test_unlink_{model_var}(self):
    """Test that {model_description} record can be deleted."""
    record_id = self.{model_var}.id
    self.{model_var}.unlink()
    result = self.{ModelClass}.search([("id", "=", record_id)])
    self.assertFalse(result)
```

**Consideration:** The Jinja2 template already generates `test_create` (weak assertion: `self.assertTrue(record.id)`) and `test_read` (uses `assertIsNotNone` per field). Phase 6 expands the assertions in `test_create` to verify defaults, and adds `test_write` + `test_unlink`.

**Strategy decision:** The `odoo-test-gen` agent should REWRITE the entire test file rather than appending. This avoids the "duplicate method name" problem and ensures consistent quality. The agent already has this pattern from Phase 5 (Write tool for full rewrite).

### Pattern 7: Access Rights Tests

**What:** Test that a user with the module's User group CAN access, and a user WITHOUT any relevant group CANNOT.

**Required imports:**
```python
from odoo.tests.common import TransactionCase
from odoo.exceptions import AccessError, ValidationError
```

**`AccessError` import path (Odoo 17.0):** `from odoo.exceptions import AccessError`
This path is unchanged since Odoo 14.0. No breaking change in 17.0.

**Confirmed pattern:**
```python
# Source: knowledge/testing.md - Access Rights Tests section

@classmethod
def setUpClass(cls):
    super().setUpClass()
    cls.{ModelClass} = cls.env["{model_name}"]
    # Create a user with module User group
    cls.user_{role} = cls.env["res.users"].create({
        "name": "Test User",
        "login": "test_user_{module}@test.com",
        "groups_id": [(6, 0, [
            cls.env.ref("{module_name}.group_{module_name}_user").id,
        ])],
    })

def test_{role}_can_create(self):
    record = self.{ModelClass}.with_user(self.user_{role}).create({
        "name": "Access Test Record",
    })
    self.assertTrue(record.id)

def test_no_group_cannot_create(self):
    # Create user with only internal user group (not module group)
    no_group_user = self.env["res.users"].create({
        "name": "No Access User",
        "login": "no_access_{module}@test.com",
        "groups_id": [(6, 0, [self.env.ref("base.group_user").id])],
    })
    with self.assertRaises(AccessError):
        self.{ModelClass}.with_user(no_group_user).create({
            "name": "Should Fail",
        })
```

**Important note on `with_user()`:** `with_user()` changes the environment user for that operation. Admin bypasses ALL ACLs — always use a non-admin test user. This is the #1 mistake in Odoo access rights testing.

**Confidence:** HIGH — pattern taken directly from `knowledge/testing.md` Access Rights Tests section.

### Pattern 8: Workflow State Transition Tests

**What:** When `workflow_states` is populated in the spec for a model, generate tests that call the action methods and assert state changes.

**Trigger:** `model.get("workflow_states", [])` in the spec — same key the view template already uses.

**Pattern:**
```python
# Source: knowledge/testing.md - Workflow Tests section

def test_action_{next_state}(self):
    """Test transition from {current_state} to {next_state}."""
    self.assertEqual(self.{model_var}.state, "{initial_state}")
    self.{model_var}.action_{next_state}()
    self.assertEqual(self.{model_var}.state, "{next_state}")
```

**Method name convention:** `action_{state_key}` — matches what `odoo-model-gen` generates for workflow transitions (confirmed from Phase 5 generate.md Task B spec).

**Edge case:** Only generate workflow tests if the model has a `state` field AND `workflow_states` is non-empty. Degenerate case (only one state) gets no transition tests.

**Confidence:** HIGH — `workflow_states` key confirmed in spec.md spec schema; action method naming confirmed from generate.md Wave 1 prompt.

### Pattern 9: odoo-security-gen Agent Placement in generate.md

**Decision:** `odoo-security-gen` runs in **Wave 1** alongside `odoo-model-gen`, not in Wave 2.

**Rationale:**
- Security generation reads only `spec.json` — it does not need completed model files
- Record rules require knowing which models have `company_id` — available from spec.json
- Wave 2 (`odoo-view-gen` + `odoo-test-gen`) needs security files to exist so access rights tests can reference `env.ref("{module}.group_{module}_user")`
- Therefore: security must be in Wave 1 (or Pass 1 Jinja2 pre-processing)

**Better alternative:** Since security is deterministic (no method bodies, no AI reasoning needed), keep it in the **Jinja2 renderer (Pass 1)** rather than adding an agent. The `security_group.xml.j2` template already handles groups. Phase 6 only needs to extend the template with the conditional record-rules block.

**This means `odoo-security-gen` does NOT need to be added to `generate.md` as a new Wave task.** Instead:
1. Extend `security_group.xml.j2` with the conditional record-rules block
2. Extend `_build_model_context()` in `renderer.py` with `has_company_field` boolean
3. The renderer handles all security deterministically — no agent needed

**`odoo-security-gen` agent role after Phase 6:** Activates as a standalone agent for users who want to customize security beyond the generated defaults (e.g., `/odoo-gen:security` for advanced use cases). Not called by `generate.md` automatically.

### Anti-Patterns to Avoid

- **Testing as admin:** `self.env` runs as admin by default. Admin bypasses ALL ACLs. Access rights tests run as admin are false negatives — they always pass even when security is broken.
- **Direct state assignment in workflow tests:** `self.record.state = "done"` bypasses `@api.constrains`, side effects, and tracking. Always call `action_{state}()`.
- **Flat group hierarchy:** Omitting `implied_ids` on Manager group means managers must be manually assigned both groups.
- **`user.company_id` in record rules:** Single-valued; breaks when user belongs to multiple companies. Always use `company_ids`.
- **CSV before XML in manifest:** ACLs load before group definitions → "External ID not found" install failure.
- **Empty `group_id:id` in CSV:** Grants access to portal and public users — security hole.
- **`(6, 0, [...])` vs `(4, id)` in `groups_id`:** Use `(6, 0, [ids])` to SET the groups list. Use `(4, id)` to ADD one group to existing list. In test `setUpClass`, use `(6, 0, [...])` to define the complete group set.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Access control enforcement | Custom Python permission checker | `ir.model.access.csv` rows + Odoo ACL engine | Odoo's ACL engine is transactional, cached, and multi-layer |
| Row-level security | Python `@api.model` filtering | `ir.rule` records with `domain_force` | ORM applies rules at DB level; Python filtering misses direct ORM calls |
| Group inheritance | Manually tracking group membership | `implied_ids` eval chain | Odoo computes transitive group membership automatically |
| Multi-company filtering | Custom domain injection in Python methods | `ir.rule` with `company_ids` domain | Record rules apply to ALL reads/writes/creates/unlinks atomically |
| Test user creation | Real Odoo user with email verification | `env["res.users"].create({"login": ..., "groups_id": ...})` in setUpClass | Test users in `TransactionCase` are auto-rolled-back; no email needed |
| Test isolation | Database locks, manual rollbacks | `TransactionCase` (auto-rollback per test) | Odoo handles isolation — trust the framework |
| Permission assertion | Try/except in tests | `with self.assertRaises(AccessError)` | Standard Python unittest pattern; clear failure messages |

**Key insight:** Odoo's security layer is the most battle-tested part of the codebase. Every custom solution adds a gap between Odoo's enforcement and your check. The generated code must USE Odoo's mechanisms, not replicate them.

## Common Pitfalls

### Pitfall 1: Wrong `model_id:id` Value in CSV
**What goes wrong:** Install fails with "External ID not found" for the model reference.
**Why it happens:** Developer writes `model_library.book` (dot preserved) instead of `model_library_book` (dot replaced with underscore).
**How to avoid:** The `model_ref` Jinja2 filter in `renderer.py` handles this correctly. The existing `access_csv.j2` already uses `{{ model.name | model_ref }}`. Do not bypass this filter.
**Warning signs:** Odoo install error containing "External ID not found" + the model name.

### Pitfall 2: Wrong Load Order in `__manifest__.py`
**What goes wrong:** Module installs with "External ID not found" for `group_library_user` referenced in the CSV.
**Why it happens:** `security/ir.model.access.csv` is loaded before `security/security.xml` in the manifest `data` list.
**How to avoid:** The `_compute_manifest_data()` function in `renderer.py` already enforces correct order: `security.xml` then `ir.model.access.csv`. Do not reorder.
**Warning signs:** Install error mentioning a group external ID rather than a model external ID.

### Pitfall 3: Testing as Admin Masks ACL Bugs
**What goes wrong:** Access rights tests pass even though security is broken for real users.
**Why it happens:** `self.env` in tests runs as the admin user. Admin bypasses ALL `ir.model.access.csv` rules.
**How to avoid:** Always create test users with specific groups and use `self.Model.with_user(test_user).create(...)` for access rights tests.
**Warning signs:** Access rights tests that never raise `AccessError` even when group is removed.

### Pitfall 4: `implied_ids` Missing on Manager Group
**What goes wrong:** Users in Manager group cannot see records they should access.
**Why it happens:** Manager group was defined without `implied_ids` pointing to User group. The ACL for User allows access, but Manager is a separate group not linked to User.
**How to avoid:** Manager `implied_ids` must reference User group. Existing template is correct — do not remove this field.
**Warning signs:** Admin can see records but Manager user gets "Access Denied" in UI.

### Pitfall 5: `user.company_id` Instead of `company_ids` in Record Rules
**What goes wrong:** Multi-company users see incorrect records or get silent access denied.
**Why it happens:** `user.company_id` is the CURRENT active company (single value). When user switches companies or is assigned to multiple, the rule breaks.
**How to avoid:** Use `company_ids` (list of all user's companies). Always include `('company_id', '=', False)` to allow records without a company field populated.
**Warning signs:** Record rule works in single-company but fails after enabling multi-company in Settings.

### Pitfall 6: Forgetting `tests/__init__.py` Imports
**What goes wrong:** Odoo silently skips test files during `/odoo-gen:validate`.
**Why it happens:** `tests/__init__.py` doesn't import the test module. Odoo discovers tests by importing `tests/__init__.py` — if the file isn't imported, it doesn't run.
**How to avoid:** The `init_tests.py.j2` template already imports all test files. When `odoo-test-gen` adds a new test file (Phase 6 doesn't add new files — it expands existing), update `__init__.py`.
**Warning signs:** Zero tests reported as discovered even though test files exist.

### Pitfall 7: Record Rule on Models Without `company_id` Field
**What goes wrong:** Template adds a record rule referencing `company_id` on a model that has no such field → install error.
**Why it happens:** Template generates record rules unconditionally rather than detecting the field.
**How to avoid:** Detect `company_id` field existence in `_build_model_context()` and pass `has_company_field` boolean to the template. Template uses `{% if model_has_company_field %}` guard.

### Pitfall 8: `(6, 0, [])` vs `(4, ref())` in XML `implied_ids`
**What goes wrong:** Groups don't inherit permissions; admin still has to manually assign both groups.
**Why it happens:** Using `(6, 0, [ref('...')])` would REPLACE the entire `implied_ids` list. The template correctly uses `(4, ref('...'))` which ADDS to the list. Both work at install time for fresh records, but `(4,...)` is the OCA standard for additive operations.
**How to avoid:** Keep the existing `eval="[(4, ref('...'))]"` pattern from `security_group.xml.j2`.
**Warning signs:** Users in Manager group don't have User permissions after a module update.

## Open Questions

1. **Does `spec.json` have a `multi_company` boolean flag?**
   - What we know: The spec schema in `spec.md` does not have an explicit `multi_company` flag. The `company_id` field presence on a model is the implicit trigger.
   - What's unclear: Should Phase 6 detect `company_id` field automatically, or should the spec schema add a `multi_company: true` flag?
   - Recommendation: Detect `company_id` field automatically in `_build_model_context()`. This is more robust — no spec schema change required, and the renderer already has field-scanning patterns (e.g., `SEQUENCE_FIELD_NAMES`). Add `has_company_field` boolean to context.

2. **Should access rights tests create BOTH user and manager test users, or just one?**
   - What we know: The standard OCA pattern tests at least two permission levels.
   - What's unclear: Creating two users per test class adds setup complexity.
   - Recommendation: Create two users in `setUpClass`: one with User group (can-create test), one with no group (cannot-create test). Manager test is implicit: if the test runs as admin (default), unlink-succeeds test is valid for "manager-equivalent". This keeps test setup manageable while covering the core security scenarios.

3. **When `odoo-test-gen` rewrites test files, does it risk losing Phase 5 computed/constraint tests?**
   - What we know: Phase 5 decided the agent rewrites entire files (Write tool). Phase 6 expands scope of the SAME agent.
   - What's unclear: The agent must preserve Phase 5-generated methods while adding Phase 6 methods.
   - Recommendation: The agent should READ the existing file first, extract all test method names to avoid duplication, then write the complete expanded file. This is already the Phase 5 agent pattern (step 3: "Read existing tests/test_{model_var}.py to avoid duplicate test method names").
